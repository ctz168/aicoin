## 三、关键对比分析

### 3.1 换算基准：1 AIC = 多少美元？

对于一个新项目（刚推出阶段），代币价值通常极低。合理估算：

| 阶段 | 假设 AIC 价格 | 0.01 AIC 实际成本 | 百万 tokens 成本 |
|------|-------------|-----------------|----------------|
| **早期（冷启动）** | $0.001 / AIC | $0.00001 | **$0.01 / 1M tokens** |
| **成长期** | $0.01 / AIC | $0.0001 | $0.10 / 1M tokens |
| **成熟期** | $0.10 / AIC | $0.001 | $1.00 / 1M tokens |
| **繁荣期** | $1.00 / AIC | $0.01 | $10.00 / 1M tokens |

### 3.3 核心问题：当前定价是否过高？

**结论：数值本身不高，但缺乏分级定价，定价体系不够精细。**

#### 问题一：没有区分输入/输出 token

当前 AICoin 统一按总 token 数计费（0.01 AIC/1K），但行业惯例是输出 token 比输入贵 3-5 倍：

| 平台 | 输入/输出价格比 |
|------|--------------|
| OpenAI (GPT-4o) | 1 : 4 |
| Anthropic (Claude 4 Sonnet) | 1 : 5 |
| Together AI (Llama 70B) | 1 : 1.14 |
| Groq (Llama 70B) | 1 : 1.34 |
| **AICoin（当前）** | **1 : 1（统一）** |

#### 问题二：没有区分模型大小定价

当前所有 Basic 档位的模型统一 0.01 AIC/1K tokens，但实际上 70B 模型的算力消耗是 7B 模型的 10 倍以上。虽然代码中 SUPPORTED_MODELS 有区分，但 BURN_RATES 只按优先级层级，没有按模型参数量调整。

#### 问题三：没有启动期促销定价

作为一个刚推出的去中心化网络，早期需要用户冷启动。当前 0.01 AIC/1K 的价格对于尚无交易市场的代币来说，心理门槛偏高。

## 四、建议定价方案

### 4.1 新定价核心原则

1. **启动期超低价**：吸引用户尝试，建立生态
2. **模型分级定价**：小模型便宜，大模型贵（反映真实算力成本）
3. **输入/输出分离计费**：符合行业惯例，输出价格更高
4. **代币单位更小**：降低心理门槛

### 4.2 建议新价格表

#### 按模型分级（输入 / 输出分离）

| 模型 | 参数量级别 | 输入价格 | 输出价格 | 对标参考 |

| aicoin-mistral-7b | 小模型 (≤7B) | 0.001 AIC / 1K | 0.003 AIC / 1K | 比 Groq 便宜 ~50x |

#### 优先级层级（倍率调整）

| 档位 | 倍率 | 说明 |
|------|------|------|
| **Basic** | 1.0x | 标准优先级，均衡路由 |
| **Premium** | 2.0x | 高优先级，延迟优先路由 |
| **Priority** | 3.0x | 最高优先级，可使用繁忙节点 |

> Premium 从原来的 5x 降到 2x，Priority 从 10x 降到 3x。原因是溢价过高会导致用户直接选择其他平台。

### 4.4 GPU 算力成本测算

以 A100 80GB 为例，验证定价合理性：

| 参数 | 值 |
|------|-----|
| A100 租赁成本 | ~$1.20 / 小时 |
| Llama 70B 推理速度 | ~30 tokens / 秒 |
| 每小时产出 | ~108K tokens |
| 每小时 GPU 成本 | $1.20 |
| 每百万 tokens 算力成本 | ~$11.1 |

> 说明：早期阶段 AIC 代币价值远低于 GPU 成本覆盖线，这是正常的。通过挖矿奖励补贴节点运营，待网络成熟后代币升值自然覆盖成本。这正是"先烧钱获取用户，后通过规模效应盈利"的典型 Web3 经济模型。

## 六、实施建议

1. **`core/api_gateway.py`**：
   - 修改 `BURN_RATES` 为模型分级 + 输入输出分离

   - 添加模型定价配置

3. **`README.md`**：

### 6.2 向后兼容

- 旧版 API 请求不区分输入输出 token，系统自动按输入价格计费
- 旧的固定 tier 定价保留为默认值，可通过治理提案切换到新的分级定价

### 6.3 治理灵活性

所有价格参数均支持通过 `PARAM_CHANGE` 治理提案调整，无需硬编码升级。社区可以根据网络运行状况动态优化定价。
