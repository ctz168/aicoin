## 一、问题分析

### 1.1 节点动态波动 (Churn) 对模型稳定性的影响

在去中心化网络中，节点随时可能上线和下线（矿工开关客户端、网络中断、硬件故障等），这对模型推理稳定性造成三大挑战：

| 场景 | 原架构问题 | 影响 |
|------|-----------|------|
| 节点突然下线 | 无熔断机制，请求被路由到已下线节点 | 请求失败、超时 |
| 节点频繁上下线 | 每次请求重新选节点，无会话保持 | 上下文重建开销大 |
| 多节点同时下线 | 仅靠 fallback 重试，没有集群级别的容错 | 级联失败 |
| 节点恢复后 | 直接恢复接收新请求，无预热 | 冷启动延迟 |

**核心风险**：原架构中 `find_best_node()` 每次请求独立选择节点，没有考虑节点的历史行为、健康状态和会话上下文，导致：
- 同一对话被分散到不同节点，上下文反复重建
- 故障节点未被及时隔离，持续浪费路由时间
- 节点恢复后没有渐进恢复机制，冷启动导致首请求超时

### 1.2 多节点 + 小模型的算力冗余

原架构将所有能运行该模型的节点平等对待，当小模型（如 7B）有大量节点在线时：

| 场景 | 原架构行为 | 问题 |
|------|-----------|------|
| 100 个节点都能跑 7B | 所有节点平等竞争请求 | 99% 的节点长期空闲 |
| 请求量低 | 每个请求都在全部节点中选最优 | 路由决策开销 >> 实际推理开销 |
| 节点规格不一 | 只看评分权重，不区分大/小 GPU | 小 GPU 节点抢不到大模型请求，大 GPU 浪费 |

**核心浪费**：7B 模型单节点即可完整推理，100 个节点意味着 99% 的算力被浪费在等待状态。原架构缺乏"按需分配"的概念，每个模型平等使用全部节点池。

### 1.3 大模型单节点装不下的问题

当模型参数量超过单个节点的显存容量时（如 70B 模型需要 ~140GB FP16 权重），原架构完全无法处理：

| 模型 | FP16 权重 | 单节点 VRAM (RTX 4090) | 原架构可行性 |
|------|----------|------------------------|--------------|
| 7B | ~14 GB | 24 GB | ✅ 正常 |
| 13B | ~26 GB | 24 GB | ⚠️ 勉强可运行 |
| 34B | ~68 GB | 24 GB | ❌ 无法运行 |
| 70B | ~140 GB | 24 GB | ❌ 完全不可行 |
| 120B+ | ~240+ GB | 24 GB | ❌ 完全不可行 |

**核心瓶颈**：原架构的 `STANDALONE` 模式假设每个节点都持有完整模型权重，这在 34B 以上的模型上是不可行的。

## 二、混合调度架构设计

### 2.1 总体架构

```
┌─────────────────────────────────────────────────────────┐
│                    HybridScheduler (混合调度器)                   │
│                                                            │
│  ┌──────────┐  ┌───────────────┐  ┌──────────────┐            │
│  │ Circuit  │  │   Session     │  │   Cluster    │            │
│  │ Breaker  │  │   Affinity    │  │   Manager    │            │
│  └────┬─────┘  └──────┬───────┘  └──────┬───────┘            │
│       │               │                  │                     │
│  ┌────┴───────────────┴──────────────────┴──────────┐            │
│  │              ModelClassifier (模型分级器)              │            │
│  │  TINY(≤8B) → STANDALONE (1节点)                   │            │
│  │  SMALL(9-15B) → STANDALONE (1节点)                │            │
│  │  MEDIUM(16-40B) → TENSOR_PARALLEL (2-4节点)        │            │
│  │  LARGE(41-72B) → TENSOR_PARALLEL (4节点)            │            │
│  │  XLARGE(73-120B) → PIPELINE_PARALLEL (4-8节点)      │            │
│  │  MASSIVE(>120B) → HYBRID (8+节点)              │            │
│  └──────────────────────────────────────────────────┘            │
└─────────────────────────────────────────────────────────┘
```

### 2.2 模型分级标准

根据模型参数量自动分级，决定推理模式和所需最小节点数：

| 级别 | 参数量 | 推理模式 | 最小节点数 | 单节点最小 VRAM | 适用场景 |
|------|--------|---------|-----------|--------------|---------|
| TINY | ≤ 8B | STANDALONE | 1 | 8 GB | 高频轻量请求 |
| SMALL | 9-15B | STANDALONE | 1 | 16 GB | 中等复杂度 |
| MEDIUM | 16-40B | TENSOR_PARALLEL | 2 | 24 GB | 需要分片 |
| LARGE | 41-72B | TENSOR_PARALLEL | 4 | 24 GB | 大模型推理 |
| XLARGE | 73-120B | PIPELINE_PARALLEL | 4-8 | 24 GB | 超大模型 |
| MASSIVE | >120B | HYBRID | 8+ | 24 GB | 极端场景 |

### 2.3 四种推理模式详解

#### STANDALONE（单机独立）

```
请求 → [节点 A (完整模型权重)] → 响应

特点: 
- 单节点持有完整模型权重，独立完成推理
- 适用于 7B、13B 等小模型
- 优势: 延迟最低、实现简单、无通信开销
- 防冗余: 严格只选 1 个节点，其余节点空闲可服务其他请求
```

#### TENSOR_PARALLEL（张量并行）

```
请求 → [分词器/嵌入] → [分片调度器]
                            ↓
            ┌──────────────┬──────────────┐
            │  Node 0      │  Node 1      │  ...  │  Node N-1
            │  Layer 0-31  │  Layer 32-63  │      │  Layer ...
            │  (权重分片0) │  (权重分片1) │      │  (分片N-1)
            └──────────────┴──────────────┘
                            ↓
            [All-Reduce 聚合] → [生成器] → 响应

特点:
- 同层权重均匀切分到多个节点
- 所有节点同时计算，通信量与分片数成正比
- 适用于 34B-72B 模型
- 通信模式: All-Reduce (每层一次全局同步)
- 网络要求: 节点间低延迟 (<10ms RTT)
```

#### PIPELINE_PARALLEL（流水线并行）

```
请求 → [嵌入层] → [Transformer Layer 0-15] → [Transformer Layer 16-31] → [LM Head] → 响应
          Node 0          Node 1              Node 2          Node 3

特点:
- 模型按层分段分配到不同节点
- 每个节点独立处理自己分配的层
- 节点间通信仅在层边界 (微批数据传递)
- 适用于 73B-120B 模型
- 通信要求: 节点间中延迟 (<50ms RTT 可接受)
- Pipeline Bubble: 节点间负载不均时会有空闲等待
```

#### HYBRID（混合并行）

```
请求 → [嵌入+分词]
            ↓
    ┌───────┬───────┬───────┐
    │  Stage 0          │  Stage 1          │  Stage 2
    │  Node 0          │  Node 0,1         │  Node 2,3
    │  TP=2, PP=1     │  TP=2, PP=1       │  TP=1, PP=1
    │  (嵌入+前8层)   │  (中间16层)      │  (后8层+LM Head)
    └───────┴───────┴───────┘

特点:
- 张量并行 + 流水线并行的混合
- 超大模型 (>120B) 的唯一可行方案
- 兼顾通信量和计算效率
- 实现复杂度最高
```

## 三、核心组件设计

### 3.1 CircuitBreaker（熔断器）

**解决问题**: 节点动态波动导致的级联失败

**状态机**:

```
CLOSED ──(连续失败≥5次)──→ OPEN ──(等待30s)──→ HALF_OPEN ──(探测成功≥2次)──→ CLOSED
  ↑                                                                  │
  └──────────(正常放行)──────────────────────────────────────────────┘
```

**参数设计**:

| 参数 | 默认值 | 说明 |
|------|--------|------|
| failure_threshold | 5 | 连续失败多少次触发熔断 |
| recovery_timeout | 30s | 熔断后等待多久才试探 |
| half_open_max_calls | 3 | 半开状态最多放行多少试探请求 |
| success_threshold | 2 | 半开状态连续成功多少次才恢复 |

**为什么不用简单的超时替代熔断**:

| 方案 | 节点临时故障 | 节点持续故障 | 节点间歇性故障 |
|------|------------|------------|-------------|
| 简单超时 | ⚠️ 浪费等待时间 | ✅ 最终超时 | ❌ 反复超时/恢复 |
| 熔断器 | ✅ 快速跳过 | ✅ 持续熔断 | ✅ 探测性恢复 |

### 3.2 SessionAffinity（会话亲和性）

**解决问题**: 同一对话被分散到不同节点导致上下文重建

**三种策略**:

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| STRONG | 仅节点故障时才切换 | 多轮对话、上下文敏感 |
| WEAK | 每 10 次请求重新评估 | 独立 API 调用（默认） |
| NONE | 每次重新选择节点 | 无状态的单次请求 |

**节点下线处理流程**:

```
节点 A 下线
    │
    ├─→ 1. CircuitBreaker: 将 A 标记为 OPEN
    │
    ├─→ 2. SessionAffinity: 查找绑定到 A 的所有会话
    │
    ├─→ 3. 对每个受影响会话:
    │       ├─→ 取消绑定 (unbind)
    │       ├─→ 选择替代节点 B
    │       └─→ 重新绑定到 B
    │
    └─→ 4. ClusterManager: 重建受影响的集群
```

### 3.3 NodeCluster（推理集群）

**解决问题**: 大模型需要多节点协作 + 小模型算力浪费

**集群效率计算**:

```
efficiency = 0.3 × 节点利用率 + 0.4 × 成功率 + 0.3 × 负载均衡

节点利用率 = min_nodes / actual_nodes  (越接近 1.0 越好)
成功率 = 成功请求 / 总请求
负载均衡 = 1 - std(VRAM_available) / mean(VRAM_available)  (方差越小越好)
```

**自适应缩容规则**:

| 条件 | 动作 |
|------|------|
| STANDALONE 模式且平均节点数 > 1 | 缩容到 1 个节点 |
| 集群效率 < 0.3 且节点数 > min_nodes | 缩容到 min_nodes |
| 集群空闲 > 5 分钟 | 回收集群，释放资源 |
| 节点下线后节点数 < min_nodes | 标记不活跃，等待重建 |

### 3.4 HybridScheduler（混合调度器）

**调度决策流程**:

```
API 请求到达 (model_name, session_key)
    │
    ├─→ 1. ModelClassifier.get_profile(model_name)
    │       → 获取: tier, mode, min_nodes, min_vram
    │
    ├─ 2. SessionAffinity.get_affinity(session_key, model_name)
    │       → 有绑定 → preferred_node = 绑定的节点
    │       → 无绑定 → preferred_node = None
    │
    ├─→ 3. CircuitBreaker: 过滤被熔断的节点
    │
    ├─→ 4. ClusterManager.find_or_create_cluster(...)
    │       → 复用已有集群 或 组建新集群
    │
    ├─→ 5. 执行推理 (with fallback)
    │       ├─ STANDALONE: 在 leader 节点执行
    │       └─ PARALLEL: 先 leader → 失败则按 rank 顺序重试
    │
    └─→ 6. 记录结果
        ├─ 成功 → 更新会话亲和性 + 熔断器记录成功
        └─ 失败 → 熔断器记录失败 + 必要时标记集群不活跃
```

## 四、奖励分配方案

### 4.1 STANDALONE 模式

```
API 收入: 100 AIC
    │
    ├─→ 金库: 20 AIC (固定)
    └─→ 节点奖励: 80 AIC
          │
          └─→ [执行节点 A] → 80 AIC (全部)
```

### 4.2 TENSOR_PARALLEL 模式

```
API 收入: 100 AIC
    │
    ├─→ 金库: 20 AIC
    └─→ 节点奖励池: 80 AIC
          │
          ├─→ Leader (Node 0): 5% 协调费 + 按权重分片比例分配
          ├─→ Worker (Node 1): 按权重分片比例分配
          ├─→ Worker (Node 2): 按权重分片比例分配
          └─→ Worker (Node N): 按权重分片比例分配

示例 (4节点均匀分片):
    Leader:  80 × 5% + 80 × 25% = 24 AIC
    Worker:  80 × 25% = 20 AIC (×3)
    总计:   24 + 20×3 = 84 AIC  ✅ (80 + 4)
```

### 4.3 PIPELINE_PARALLEL 模式

```
API 收入: 100 AIC
    │
    ├─→ 金库: 20 AIC
    └─→ 节点奖励池: 80 AIC
          │
          ├─→ Leader (Node 0): 30% 协调费 + 按层分配比例
          └─→ Workers: 按层分配比例

示例 (4节点均匀分层):
    Leader:  80 × 30% + 80 × 23% = 42 AIC
    Worker:  80 × 23% = 18 AIC (×3)
    总计:   42 + 18×3 = 96 AIC  ✅ (80 + 16)
```

## 五、配置参数汇总

### 5.1 config.py 新增配置项

```python
# === 混合推理 ===
hybrid_inference_enabled: bool = True          # 是否启用混合推理
hybrid_routing_strategy: str = "HYBRID"        # 路由策略
cb_failure_threshold: int = 5                    # 熔断器: 连续失败阈值
cb_recovery_timeout: float = 30.0              # 熔断器: 恢复时间(秒)
cb_half_open_max_calls: int = 3              # 熔断器: 半开试探次数
cb_success_threshold: int = 2                  # 熔断器: 恢复所需成功次数
affinity_strategy: str = "weak"                # 会话亲和性: strong/weak/none
affinity_ttl: float = 600.0                    # 会话亲和性 TTL (秒)
cluster_gc_interval: float = 60.0              # 集群回收检查间隔(秒)
cluster_max_idle_seconds: float = 300.0         # 集群最大空闲时间(秒)
```

### 5.2 模型分级自动识别

系统内置 14 个模型的分级配置，未注册模型自动估算：

1. 从模型名提取参数量（正则匹配 `(\d+)b`）
2. 根据参数量自动分级
3. 自动选择推理模式和最小节点数

## 七、性能预期

| 指标 | 原架构 | 混合架构 | 改善幅度 |
|------|--------|----------|---------|
| 节点故障恢复延迟 | 30s (等待超时) | 30s (熔断) + 即时重路由 | 故障影响范围从全局缩小到单集群 |
| 同一对话上下文切换率 | 100% (每次重选) | <10% (弱亲和性) | 10x 降低 |
| 7B 模型算力利用率 | ~1% (100节点只1个在工作) | ~70-80% (按需1-3节点) | 70-80x 提升 |
| 70B 模型可行性 | ❌ 不可用 | ✅ 4节点张量并行 | 从0到可用 |
| 路由决策延迟 | O(N) 全量评分 | O(K) 复用已有集群 | N >> K 时显著改善 |
| 级联失败风险 | 高 (全节点共享) | 低 (集群隔离) | 故障隔离 |
